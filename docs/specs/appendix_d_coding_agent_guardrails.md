# Appendix D: Coding Agent Guardrails

## Purpose

This appendix defines the guardrails, rules, and operating constraints for AI coding agents (specifically Claude Code and GLM-5.1) when implementing CortexOS. These guardrails exist to prevent systematic errors that are common when AI agents generate code without explicit constraints. Every rule in this document is a reaction to a specific failure mode observed in AI-assisted development.

## Scope

These guardrails apply to all code generated by AI coding agents for the CortexOS project, regardless of which agent produces the code. They apply to Rust backend code, TypeScript frontend code, configuration files, test files, and documentation.

---

## D.1. Hard Rules

Hard rules must NEVER be violated. Violation of a hard rule is a blocking issue that must be fixed before the code can be merged. There are no exceptions.

### D.1.1. No Client-Side Authorization

**Rule**: The browser/client must never be the source of truth for authorization decisions. The server (Rust backend) must enforce all authorization, authentication, and permission checks. The client may use permission state for UI rendering (e.g., hiding a button) but must not rely on client-side checks for security.

**Why**: Client-side state can be modified by the user or by malicious scripts. Any authorization decision made only on the client is bypassable.

**Enforcement**:
- Every permission check must have a corresponding server-side enforcement point.
- PR reviews must verify that no authorization logic exists only on the client.
- The test suite must include tests that verify server-side rejection of unauthorized requests.

**Example of violation**:
```typescript
// WRONG: Client checks permission and skips server call
if (user.hasPermission("ai.file_access")) {
  await aiService.readFile(fileId);
}
```

**Example of compliance**:
```typescript
// CORRECT: Client attempts operation, server enforces permission
try {
  const result = await aiService.readFile(fileId);
  // success
} catch (e) {
  if (e.code === "PERMISSION_DENIED") {
    // show user-facing error
  }
}
```

### D.1.2. No Policy Bypass

**Rule**: No code path may bypass the policy service (cortex-policy). If the policy service defines a check for an operation, that check must be invoked for every execution of that operation, regardless of the caller (first-party app, third-party app, system service, or AI runtime).

**Why**: Policy bypasses create security holes that are difficult to detect because they work correctly in normal testing but fail under adversarial conditions.

**Enforcement**:
- Every operation that requires a policy check must have the check in a single, central location (not duplicated at each call site).
- PR reviews must verify that no "fast path" skips policy checks.
- Integration tests must verify that policy checks are enforced for all callers.

### D.1.3. No Provider Lock-In

**Rule**: The core AI runtime (cortex-ai) must not contain any provider-specific logic outside of provider adapter implementations. The routing engine, fallback logic, budget tracking, and privacy preprocessing must be provider-agnostic. No provider may be treated as a "default" or "special" provider in core logic.

**Why**: Provider lock-in makes it difficult to add new providers, creates brittle code that breaks when a provider changes its API, and violates the core product principle that users choose their LLM.

**Enforcement**:
- All provider-specific code must be isolated in adapter modules under `cortex-ai/src/adapters/`.
- The routing engine must treat all providers identically.
- No provider name may appear in the routing logic as a special case.
- PR reviews must verify that new provider integrations follow the adapter pattern.

### D.1.4. No Hidden First-Party Privileges

**Rule**: First-party apps (Settings, Calculator, Text Editor, Notes, File Manager, Media Viewer, Terminal-lite, Clock Utilities) must not have hidden privileges that are not available to third-party apps through the same permission system. If a first-party app needs a capability, that capability must be declared in the app manifest and granted through the same policy system.

**Why**: Hidden privileges create an unfair platform and make the permission system untrustworthy. Third-party developers must be able to understand and request the same capabilities.

**Enforcement**:
- First-party app manifests must declare all required permissions, the same as third-party apps.
- The policy system must not have a "first-party bypass" flag.
- PR reviews must verify that first-party apps do not use private APIs that bypass the permission system.
- The only difference between first-party and third-party apps is that first-party apps are pre-installed and their permissions are pre-granted during system initialization. The pre-grant uses the same mechanism as user-initiated grants.

### D.1.5. No Silent Conflict Resolution

**Rule**: When a conflict occurs (concurrent file edits, settings race conditions, permission revocation during operation), the system must not silently resolve the conflict without informing the user. The conflict must be surfaced, and the resolution strategy must be explicit and documented.

**Why**: Silent conflict resolution leads to data loss, unexpected behavior, and user confusion. Users must know when their data or state has been modified by the system.

**Enforcement**:
- Every conflict scenario defined in the specs must have an explicit resolution strategy.
- The resolution strategy must include user notification.
- PR reviews must verify that conflict cases are handled with user-visible feedback.

### D.1.6. No Untyped Events

**Rule**: All events on the system command bus (spec 10) must have a typed payload. No event may use untyped maps (e.g., `HashMap<String, Any>` or `serde_json::Value`) as its payload type. Every event must have a named struct with typed fields.

**Why**: Untyped events lead to runtime errors, make refactoring dangerous, and prevent compile-time verification of event contracts.

**Enforcement**:
- Every event type must be defined as a Rust struct with typed fields.
- All event structs must derive or implement `Serialize`, `Deserialize`, `Clone`, and `Debug`.
- Events must be registered in a central event registry with unique type names.
- PR reviews must verify that no event uses `serde_json::Value` as a payload field.

**Example of violation**:
```rust
// WRONG: Untyped event payload
Event {
    event_type: "file.created",
    payload: HashMap::from([("path", "/foo.txt"), ("size", "1024")]),
}
```

**Example of compliance**:
```rust
// CORRECT: Typed event payload
#[derive(Serialize, Deserialize, Clone, Debug)]
struct FileCreatedEvent {
    path: VirtualPath,
    size: u64,
    file_id: FileId,
    created_at: DateTime<Utc>,
}
```

### D.1.7. No UI-Only Definition of Done

**Rule**: A feature is not done when the UI renders correctly. A feature is done only when: the UI renders correctly, the server-side logic is implemented, error handling is complete, permissions are enforced, tests pass, and documentation exists. See appendix C for the complete definition of done.

**Why**: AI agents tend to prioritize visible output (UI) over invisible infrastructure (server logic, error handling, tests, permissions). This rule prevents shipping features that look complete but are not.

**Enforcement**:
- PRs must include evidence of all definition-of-done criteria, not just screenshots.
- PR templates include the feature checklist from appendix C.
- PR reviews must verify all checklist items, not just visual correctness.

## D.2. Soft Rules

Soft rules should be followed. Deviation from a soft rule requires a comment in the code explaining why the deviation is necessary and what alternative approach was chosen.

### D.2.1. Prefer Explicit Over Implicit

- Prefer explicit type annotations over type inference for public APIs.
- Prefer explicit error types over generic `Box<dyn Error>`.
- Prefer explicit configuration over convention-based configuration.
- Prefer explicit dependency injection over global state or singletons.
- Prefer explicit field names in struct initialization over positional constructors.

### D.2.2. Prefer Typed Over Untyped

- Prefer Rust enums over string constants for status codes and categories.
- Prefer newtypes over raw primitives for domain concepts (e.g., `AppId(String)` rather than `String`).
- Prefer typed builder patterns over functions with many optional parameters.
- Prefer typed error enums over string error messages.

### D.2.3. Prefer Server-Side Validation Over Client-Side Validation

- All input validation must happen on the server, even if it also happens on the client.
- Client-side validation is allowed for user experience (immediate feedback) but must not be the only validation.
- Server-side validation must be the authoritative check.
- Error messages from server-side validation must be user-friendly (not raw validation error dumps).

### D.2.4. Prefer Composition Over Inheritance

- Rust has no inheritance. Use traits for shared behavior.
- Prefer trait objects (`dyn Trait`) over enum dispatch when the set of implementations is open-ended.
- Prefer enum dispatch when the set of implementations is fixed and known.
- Prefer composition (structs containing other structs) over deep trait hierarchies.

### D.2.5. Prefer Small, Focused Functions

- Functions should do one thing.
- Functions longer than 50 lines should be refactored unless the complexity is inherent.
- Functions should have descriptive names that make comments unnecessary.

### D.2.6. Prefer Immutable Data Over Mutable State

- Prefer `&self` methods over `&mut self` methods when possible.
- Prefer returning new values over mutating in place when the data is shared.
- Use interior mutability (`RefCell`, `Mutex`) only when the ownership model requires it, not as a default.

## D.3. Testing Requirements for Agents

### D.3.1. Mandatory Test Accompaniment

Every implementation PR must include tests. There are no exceptions.

- If the change adds a new public function, it must have unit tests.
- If the change adds a new API endpoint, it must have integration tests.
- If the change adds a new user flow, it must have an E2E test (or a test plan that is tracked as an issue).
- If the change fixes a bug, it must include a regression test that would have caught the bug.

### D.3.2. Test Quality Requirements

- Tests must test behavior, not implementation details.
- Tests must not be trivially satisfied (e.g., `assert_eq!(2 + 2, 4)` is not a valid test for business logic).
- Tests must include assertions. A test that runs code without assertions is not a valid test.
- Tests must be deterministic. A test that fails intermittently is a bug in the test, not in the code.
- Tests must clean up their state. A test that leaves side effects is a bug in the test.

### D.3.3. What Tests to Write

When implementing a feature, the agent must write tests for:

1. **Happy path**: the primary use case works correctly.
2. **Error paths**: each defined error case produces the correct error.
3. **Boundary conditions**: empty inputs, maximum-size inputs, edge cases.
4. **Authorization**: unauthorized access is rejected (when applicable).
5. **Concurrency**: race conditions do not cause data corruption (when applicable).

## D.4. Security Checklist for Agents

Before submitting any PR, the agent must verify the following security checklist:

### D.4.1. Input Validation

- [ ] All user inputs are validated at the server boundary.
- [ ] Input validation rejects invalid data with clear error messages.
- [ ] Input validation does not silently coerce or truncate data.
- [ ] String inputs have length limits enforced.
- [ ] Numeric inputs have range limits enforced.
- [ ] File paths are canonicalized and do not allow directory traversal.
- [ ] Enum inputs are validated against the allowed set.

### D.4.2. Authentication and Authorization

- [ ] Authentication is enforced on all endpoints that require it.
- [ ] Authorization checks use the policy service, not inline checks.
- [ ] No hardcoded credentials or API keys.
- [ ] No credentials in logs, error messages, or UI.
- [ ] Session tokens are validated on every request.
- [ ] Expired sessions are rejected.

### D.4.3. Data Protection

- [ ] Sensitive data is not logged at INFO level or above.
- [ ] Sensitive data is not included in error messages shown to users.
- [ ] File contents are not accessible without permission checks.
- [ ] AI request contents are not stored beyond the audit retention period.
- [ ] User data is not shared between users.

### D.4.4. OWASP Top 10 Review

For every PR that modifies a server endpoint, the agent must consider:

1. **Injection**: Are all inputs sanitized before use in queries or commands?
2. **Broken Authentication**: Is the endpoint properly authenticated?
3. **Sensitive Data Exposure**: Is any sensitive data leaked in responses?
4. **XML External Entities**: Not applicable (CortexOS uses JSON, not XML).
5. **Broken Access Control**: Are authorization checks enforced?
6. **Security Misconfiguration**: Are default settings secure?
7. **Cross-Site Scripting**: Are all user inputs escaped before rendering in HTML?
8. **Insecure Deserialization**: Is deserialization bounded and validated?
9. **Using Components with Known Vulnerabilities**: Does `cargo audit` pass?
10. **Insufficient Logging**: Are security-relevant events logged?

### D.4.5. Never Log Secrets

This rule is absolute and has no exceptions:

- Do not log API keys, tokens, passwords, or credentials at any log level.
- Do not include credentials in error messages.
- Do not include credentials in debug output.
- Do not store credentials in plaintext in configuration files.
- Do not pass credentials through event payloads.
- When logging an AI request, log the provider, model, and token counts, not the request body or response body.

## D.5. Stop Conditions

An AI coding agent must stop and ask for human clarification in the following situations. The agent must NOT proceed autonomously.

### D.5.1. Must Stop and Ask

1. **Ambiguous spec**: The spec is unclear or contradictory, and the agent cannot resolve the ambiguity from context.
2. **Missing spec**: The feature being implemented is not covered by any spec section.
3. **Spec conflict**: Two spec sections provide conflicting guidance, and the agent cannot determine which takes precedence.
4. **Security uncertainty**: The agent is unsure whether an approach introduces a security vulnerability.
5. **Cross-subsystem impact**: The change affects a subsystem that the agent was not asked to modify, and the impact is unclear.
6. **Performance concern**: The agent's implementation may not meet performance targets, and optimization requires architectural decisions beyond the agent's scope.
7. **Dependency decision**: Adding a new external dependency requires human approval (evaluate license, maintenance status, alternative approaches).
8. **Breaking change**: The change would break a public API or data format.
9. **Database migration**: Any change to the database schema requires human review.
10. **Permission model change**: Any change to the permission or policy model requires human review.

### D.5.2. May Proceed Autonomously

The agent may proceed without asking in the following situations:

1. **Well-specified feature**: The spec is clear, complete, and unambiguous.
2. **Bug fix**: The fix is straightforward and does not change behavior beyond fixing the bug.
3. **Test writing**: Writing tests for already-implemented code.
4. **Documentation**: Writing rustdoc comments for documented public APIs.
5. **Refactoring**: Internal refactoring that does not change public APIs or behavior.
6. **Error handling**: Adding missing error handling for defined error cases.
7. **Logging**: Adding missing log statements for defined observability requirements.

### D.5.3. How to Ask

When the agent must stop and ask:

1. State the specific question or ambiguity.
2. Reference the relevant spec section number.
3. Present the options the agent has considered.
4. State the agent's recommended approach (if any).
5. Wait for human response before proceeding.

## D.6. Anti-Patterns List

The following patterns must be avoided in all CortexOS code. Each anti-pattern includes an explanation of why it is problematic.

### D.6.1. God Object / God Crate

**Anti-pattern**: A single crate or struct that handles too many responsibilities.

**Why**: Violates single-responsibility principle, makes testing difficult, creates hidden coupling, makes the codebase difficult to navigate.

**Prevention**: Each crate must have a single, well-defined responsibility as specified in its spec. If a crate needs functionality from another domain, it must depend on the appropriate crate, not absorb the functionality.

### D.6.2. Stringly-Typed Everything

**Anti-pattern**: Using strings instead of typed enums, structs, or newtypes for domain concepts.

**Examples**:
- Using `"read"` / `"write"` strings instead of a `Permission` enum.
- Using `"openai"` string instead of a `Provider` enum.
- Using raw `String` for `AppId`, `FileId`, `UserId` instead of newtypes.

**Why**: String comparisons are error-prone (typos, case sensitivity), prevent compile-time checking, and make refactoring dangerous.

**Prevention**: Use Rust's type system. Define enums for closed sets, newtypes for domain identifiers, and structs for structured data.

### D.6.3. Catch-All Error Types

**Anti-pattern**: Using `Box<dyn Error>`, `anyhow::Error`, or `String` as the error type for public APIs.

**Why**: Prevents callers from handling specific error cases, makes debugging difficult, hides the actual failure modes.

**Prevention**: Define specific error enums for each crate's public API. Use `thiserror` or equivalent for error derive. Internal code may use `anyhow` for convenience but must convert to typed errors at public API boundaries.

```rust
// WRONG: catch-all error type
pub fn get_user(id: &str) -> Result<User, anyhow::Error>

// CORRECT: specific error enum
#[derive(Debug, thiserror::Error)]
pub enum UserError {
    #[error("user not found: {0}")]
    NotFound(UserId),
    #[error("database error: {0}")]
    Database(#[from] DbError),
}
pub fn get_user(id: &UserId) -> Result<User, UserError>
```

### D.6.4. Global Mutable State

**Anti-pattern**: Using global static mutable variables (`static mut`, `lazy_static` with `Mutex`, `once_cell` with `RefCell`) for shared state.

**Why**: Makes testing difficult (cannot reset state between tests), creates hidden dependencies, introduces concurrency bugs.

**Prevention**: Pass state explicitly through function parameters or store it in a typed context struct that is initialized once and passed through the application.

### D.6.5. Silent Error Swallowing

**Anti-pattern**: Using `let _ = ...` or `.ok()` to ignore errors without logging or handling them.

**Why**: Hides failures, makes debugging extremely difficult, can lead to data loss or corruption that goes unnoticed.

**Prevention**: Every error must be either: handled explicitly, propagated to the caller, or logged with appropriate context. If an error is intentionally ignored, a comment must explain why.

```rust
// WRONG: silent error swallowing
let _ = file.delete();

// CORRECT: explicit error handling
if let Err(e) = file.delete() {
    log::warn!("Failed to delete temp file {}: {}", file.path(), e);
    // Non-critical: temp file cleanup failure should not block the operation
}
```

### D.6.6. Premature Optimization Without Measurement

**Anti-pattern**: Adding caching, pooling, or optimization before measuring that a performance problem exists.

**Why**: Adds complexity, introduces bugs (cache invalidation, stale data), and may optimize the wrong thing.

**Prevention**: Implement the straightforward solution first. Measure performance against the defined targets. Optimize only when a target is not met, and optimize the specific bottleneck identified by measurement.

### D.6.7. Magic Numbers and Strings

**Anti-pattern**: Using unnamed numeric literals or string literals in business logic.

**Why**: Makes code difficult to understand, creates maintenance burden when values need to change, prevents compile-time validation.

**Prevention**: Define constants with descriptive names for all numeric and string literals used in business logic.

```rust
// WRONG: magic number
if attempts > 3 { ... }

// CORRECT: named constant
const MAX_RETRY_ATTEMPTS: u32 = 3;
if attempts > MAX_RETRY_ATTEMPTS { ... }
```

### D.6.8. Implicit Feature Dependencies

**Anti-pattern**: Code in crate A depends on behavior in crate B without declaring a dependency in `Cargo.toml` or the spec.

**Why**: Creates hidden coupling, breaks builds when crate B changes, makes the dependency graph incorrect.

**Prevention**: All inter-crate dependencies must be explicit in both `Cargo.toml` and the subsystem spec's dependency list.

### D.6.9. Test-Implementation Coupling

**Anti-pattern**: Tests that verify implementation details (private function behavior, internal state) rather than public behavior.

**Why**: Tests break when implementation changes even if behavior is correct, discourages refactoring, provides false confidence.

**Prevention**: Test public APIs and observable behavior. Use integration tests for cross-boundary behavior. Avoid testing private functions directly.

### D.6.10. Configuration via Code Changes

**Anti-pattern**: Requiring code changes to modify behavior that should be configurable (settings, permissions, feature flags).

**Why**: Prevents runtime configuration, requires redeployment for changes, exposes implementation details to users.

**Prevention**: All user-configurable behavior must be exposed through the settings service (spec 05) or the permission system (spec 04). No behavior that the spec defines as configurable may be hardcoded.

### D.6.11. Timeout-Free External Calls

**Anti-pattern**: Making HTTP requests or other external calls without a timeout.

**Why**: External calls can hang indefinitely, blocking threads and consuming resources.

**Prevention**: Every external call (AI provider requests, any HTTP call) must have an explicit timeout. Default timeout for AI provider requests: 30 seconds. Default timeout for other HTTP calls: 10 seconds. Timeouts must be configurable via settings where appropriate.

### D.6.12. Unbounded Collections in APIs

**Anti-pattern**: API endpoints that return unbounded lists (no pagination or limit).

**Why**: Can cause memory exhaustion, slow responses, and poor UX for large datasets.

**Prevention**: All list/query API endpoints must accept a `limit` parameter with a maximum value. Default limit: 100. Maximum limit: 1000. Endpoints must return a pagination cursor when more results exist.

## D.7. Agent Operating Protocol

### D.7.1. Before Starting Implementation

1. Read the relevant spec section(s) completely.
2. Identify the scope: what is being implemented and what is NOT being implemented.
3. Identify dependencies: which other crates/subsystems does this code depend on.
4. Identify acceptance criteria: what tests must pass for this work to be considered done.
5. Identify any stop conditions that apply (section D.5).

### D.7.2. During Implementation

1. Follow the build order defined in the relevant spec.
2. Implement one feature or function at a time.
3. Write the test immediately after (or before) implementing each function.
4. Run `cargo clippy` and `cargo fmt` after each significant change.
5. Run `cargo test` after each significant change.
6. Add rustdoc comments to every public API as it is implemented.

### D.7.3. After Implementation

1. Run all quality checks: clippy, fmt, test, coverage.
2. Verify all acceptance criteria from the spec.
3. Verify all definition-of-done items from appendix C.
4. If any hard rule is violated, fix it before submitting.
5. If any stop condition applies, ask for clarification before submitting.
6. Generate the PR description referencing the spec section and acceptance criteria.

### D.7.4. Code Generation Standards

- Use idiomatic Rust. Do not generate code that looks like it was translated from another language.
- Use `Result<T, E>` for all fallible operations. Do not panic in production code.
- Use `async/await` for all I/O operations. Do not use blocking I/O.
- Use proper error propagation (`?` operator) instead of manual match on `Result`.
- Use `#[derive]` for common traits (`Debug`, `Clone`, `Serialize`, `Deserialize`) where appropriate.
- Use `Arc<Mutex<T>>` or `Arc<RwLock<T>>` for shared mutable state across async tasks.
- Use `tokio::sync` primitives (`Mutex`, `RwLock`, `Semaphore`) for async-aware synchronization.
- Do not use `std::sync::Mutex` for data that is held across `.await` points.
